# Navigation System card model

**Live page:** https://stablescouncil.org/council_navigation_system.html  
**Status:** Version 0.0.01 | Community Draft

The Stables Navigation System is the public roadmap and status map for the project. It is not a marketing page and not a final governance decision. It is a living draft that lets the community see the structure of the work.

## Hierarchy

The Navigation System uses four core levels:

1. **Stream**
2. **Group**
3. **Card**
4. **Element**

Avoid adding extra public hierarchy levels unless there is a clear reason. In particular, do not split the same card into duplicate Showcase and Demo cards. The card itself carries its app phase information.

## Streams

The three public streams are:

- **Technical Stream:** app screens, runtime flows, wallet and mint/burn flows, invest/coverage, merchant tools, infrastructure, security, previews, and production path.
- **Financial Stream:** mint/burn discipline, peg maintenance, solvency controls, coverage, liquidity, transparent protocol accounting, and merchant accepting payment network foundations for real circulation.
- **Community Stream:** communication, feedback, ambassadors, merchant growth, Council formation, and community handover.

## Dapp phase logic

The app phase boxes are filters and definitions. They should be understood as:

- **Showcase:** synthetic presentation and local simulation for explaining the product direction. It is not a live on-chain Stables product.
- **Demo:** current demo channel. It uses Minima where available, but has no official Stables production token on-chain. In-app demo assets remain demo-only.
- **Test:** future stage for real on-chain Winiwa/Wables test-token work and quests.
- **Prod:** future Council-declared production stable layer for real Minima and Stables production mechanics.

Minima is used for minting Stables and xMinima. There is no "Maximize MINIMA staking surface" product concept in the current public scope.

## Lens filters

The Navigation System includes lens filters:

- **Charter Lens:** concepts, principles, and agreements.
- **Dapp Lens:** concrete screens, flows, and practical application.

The Dapp Lens should follow the app order and practical user flow. It should not put Chat ahead of earlier banking, shop, mint, invest, merchant, communication, or dashboard surfaces unless the app structure later changes.

## Card content

Each card can include:

- App phase tags.
- Progress percentage.
- Status and blockers.
- Dependencies.
- Dashboard Measures.
- Work Evidence.
- Links to local or public pages.

Dashboard Measures are for measurable success indicators only. Non-measurable proofs, explanations, scope notes, or work artifacts belong in **Work Evidence**.

## Dashboard links

Navigation cards can link to Dashboard measures. These links are public and should be maintained when cards or metric IDs change.

If a card has no measurable success signal yet, it is acceptable to say that no Dashboard measure is linked yet. Do not force weak or fake indicators.
